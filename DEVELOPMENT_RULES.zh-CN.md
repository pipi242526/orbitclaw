# 开发规则（orbitclaw-s）

**Language / 语言**: [English](DEVELOPMENT_RULES.md) | [简体中文](DEVELOPMENT_RULES.zh-CN.md)

本分支遵循务实、轻量、可维护的工程策略。

## 0) 项目护栏（默认遵循）

1. 资源铁律：
   - 每个版本应明确内存/token/超时/队列预算上限。
   - 新功能上线前应说明预算影响；紧急情况可先发布后补齐说明。
2. 输出铁律：
   - 最终回复遵循已配置语言策略。
   - 默认不向终端用户泄露内部工具/MCP 调用细节。
   - 失败时必须提供原因与可执行修复建议。
3. 接口铁律：
   - 统一消息契约（`reply_to`、`actions`、`attachments`）。
   - 渠道适配层以协议映射为主，业务逻辑应优先收敛到核心/服务层。
4. 配置铁律：
   - 环境变量优先，明文密钥最小化。
   - 每条配置变更路径应可回滚。
5. 扩展铁律：
   - 新渠道/MCP/技能应采用插件化，尽量避免侵入核心 Agent Loop。
6. 演进铁律：
   - 每个版本应附带上游补丁审计记录（accept/reject/defer）。
7. 质量铁律：
   - 新增或修改代码必须附带聚焦测试并通过增量 lint。
   - 历史 lint 债务可暂存，但不能无感扩大。

### 例外处理

允许临时例外（例如紧急修复、发版窗口限制），但需同时记录：

1. 例外原因与影响范围
2. 回滚或缓解路径
3. 后续补齐任务（issue/commit 目标）

## 1) 优先级

1. 先保证机器人核心行为（gateway/chat/tools），再做 WebUI。
2. 稳定性优先于功能广度。
3. 保持架构简单、可组合。

## 2) 设计原则

1. 一个能力，只有一个主入口工具。
2. 优先配置驱动，避免硬编码分支。
3. 可选集成必须保持可选（依赖缺失时优雅降级）。
4. 迁移完成后及时删除兼容垫片。

## 3) 文件处理策略

1. `~/.orbitclaw/media`：入站附件（原始文件）。
2. `workspace`：处理中间文件。
3. `exports`：最终产物（报告/转换结果）。
4. 列表/删除统一走 `files_hub`（`media|exports`）。
5. 生成输出统一走 `export_file`。

## 4) 默认安全策略

1. 默认最小暴露（`allowFrom`、WebUI 路径密钥）。
2. UI 与日志不得输出敏感密钥。
3. 未经工具确认不得宣称“已完成”。

## 5) 语言与搜索

1. 最终答复语言跟随用户偏好/配置。
2. 区域性主题应支持跨语言搜索提示。
3. 工具输出可多语言，但最终回复必须遵循语言策略。

## 6) 测试与发布纪律

1. 每个新增工具或行为变化都要有聚焦测试。
2. 部署前必须跑轻量回归。
3. 提交应小而可审阅，且有清晰回滚路径。

## 7) 非目标

1. 无明确生产需求时，不引入重框架扩张。
2. 不保留“同一功能多套工具”的重复设计。
3. 迁移完成后不保留长期兼容代码。
