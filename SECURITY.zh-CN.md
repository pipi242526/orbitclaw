# 安全策略

**Language / 语言**: [English](SECURITY.md) | [简体中文](SECURITY.zh-CN.md)

## 漏洞报告

如果你发现 OrbitClaw 的安全漏洞，请：

1. **不要** 公开提 GitHub Issue。
2. 通过 GitHub 私有安全通道或联系维护者（xubinrencs@gmail.com）。
3. 提供：
   - 漏洞描述
   - 复现步骤
   - 影响范围
   - 修复建议（可选）

目标响应时间：48 小时内。

## 安全实践

### 1) API Key 管理

关键原则：不要把密钥提交到版本库。

```bash
chmod 600 ~/.orbitclaw/config.json
```

建议：
- 密钥优先放在 `~/.orbitclaw/.env` 或 `~/.orbitclaw/env/*.env`
- 生产环境使用系统密钥管理器
- 开发/生产分离密钥并定期轮换

### 2) 渠道访问控制

生产环境必须配置 `allowFrom`。

- 空的 `allowFrom` 代表允许所有用户，不适合公开部署。
- Telegram 建议按用户 ID 白名单。
- WhatsApp 建议按完整国际手机号白名单。

### 3) 命令执行安全

`exec` 工具可以执行 shell 命令，应当：

- 使用低权限专用用户运行
- 审计工具调用日志
- 不要关闭安全检查
- 不要在高敏系统上无隔离运行

默认会阻断明显危险模式（如破坏性删除、格式化磁盘等）。

### 4) 文件系统安全

虽然有路径穿越防护，仍建议：

- 使用独立运行用户
- 通过系统权限限制敏感目录
- 定期审计文件读写行为

### 5) 网络与桥接安全

- 外部 API 默认 HTTPS。
- WhatsApp bridge 默认仅监听 `127.0.0.1:3001`。
- 建议配置 `bridgeToken` 作为 Python 与 Node 之间的共享密钥。

### 6) 依赖安全

定期做依赖扫描和升级：

```bash
pip install pip-audit
pip-audit
pip install --upgrade orbitclaw-ai
```

Node 依赖（bridge）也需执行 `npm audit`。

### 7) 生产部署建议

- 尽量容器化或虚拟机隔离
- 使用专用系统用户运行
- 设置目录权限（`~/.orbitclaw` 建议 700）
- 开启日志监控和 API 额度/速率限制
- 建立备份与回滚流程

### 8) 数据隐私

- 日志可能包含敏感信息，应受保护
- LLM 提供商可见提示词，需评估其隐私政策
- 本地会保存会话/历史，需保护 `~/.orbitclaw`

### 9) 事件响应

若疑似泄露：

1. 立即吊销密钥
2. 检查未授权访问日志
3. 轮换全部凭据
4. 升级到最新版本
5. 向维护者报告

## 已内置的安全机制

- 输入校验（路径/请求长度/危险模式）
- 允许列表访问控制
- 超时与输出截断保护
- 默认 HTTPS/TLS 通信

## 当前限制（需自行补强）

- 未内置完整全局限流
- 仍有明文配置场景
- 会话过期与审计能力有限

## 部署前检查清单

- [ ] 密钥未写入代码仓库
- [ ] 配置文件权限正确（0600）
- [ ] 所有启用渠道均配置 `allowFrom`
- [ ] 进程非 root 运行
- [ ] 依赖已更新并完成安全扫描
- [ ] 日志监控与告警已配置
